## 引言

在过去的几个月里，成千上万的构建者使用 Bolt、Lovable 和 v0 等工具创建了网站和 Web 应用程序。在 X、Reddit 和 YouTube 上分享的结果看起来简直就像魔法：人们正在构建交互式的、动态的产品 —— 甚至无需编码就能产生收入。

![Twitter 上的用户分享](/images/20250212_ai_powered_web_app_builders_rise/twitter_share.png)

仅仅通过与机器人聊天就能创建功能性产品的能力，引起了技术和非技术用户的关注，导致该领域的初创公司实现了惊人的增长。公开数据显示，[Bolt](https://www.youtube.com/watch?v=MJRgjb6BYck) 和 [Lovable](https://x.com/antonosika/status/1883622036872065217) 在开始盈利后不久，就分别实现了 2000 万美元和 1000 万美元的年化收入，增长势头惊人。

一个全新的、基于大型语言模型 (LLM) 的应用技术栈正在兴起。在网站生成解决方案出现之前，如果你问一个技术人员"你的技术栈是什么？"，得到的答案会是一系列库。但这种新的应用创建方式——通过文本——引入了一个新的抽象层。对于不常写代码的用户来说，他们将使用一系列基于像素的设计生成、图像生成、网站生成和托管工具。对于这个快速增长的用户群体，新的抽象层是像素和自然语言，而不是开发人员所熟悉的 SDK 和框架。

构建者们正在使用什么产品来利用 AI 制作网站和 Web 应用程序？主要的应用场景是什么，它们吸引了哪些类型的用户？下面，我们将深入探讨这些产品的工作原理、它们的技术限制以及即将推出的功能。

## 文本到 Web 应用和网站生成器的当前格局

![市场地图](/images/20250212_ai_powered_web_app_builders_rise/current_market_map.png)

_注意：此市场地图仅限于专注于将文本转换为 Web 应用或网站的公司。还有更广泛的辅助产品 —— 用于基于 IDE 的代码生成和编辑、设计迭代 —— 我们将在下面介绍。_

这里有两个核心的产品区别：

1. ** 静态网站 vs. 应用程序生成：** 我们将静态网站定义为主要用于传达信息并具有基本交互性（例如提交联系表单或预约）的网页。应用程序更具动态性 —— 用户可以执行更复杂的操作。静态网站更容易生成，并且可以是基于模板的。动态应用程序更具状态性。这种类型的应用程序需要第三方工具（如数据库和身份验证系统）来支持 CRM、表单提交和用户登录等用例。可以想象一下小型实体企业网站（你可能可以查看信息并安排咨询）和 YouTube 等 Web 应用（你可以点赞、评论甚至上传自己的内容）之间的区别。

2. ** 使用代码导出：** 所有生成 Web 应用的产品以及一些生成网站的产品都在创建代码。但有些允许你导出此代码以在另一个应用程序中进行编辑，而另一些则不公开代码，而是要求你部署：通过他们的平台或通过集成合作伙伴的平台。对于开发人员来说，下载、查看和编辑代码的能力很重要。大多数人不会在没有在其 IDE 中进行构建或编辑的情况下发布他们在文本到 Web 应用构建器中生成的产品。对于非技术用户来说，导出代码的能力可能不太重要。

## 文本到 Web 应用产品是如何工作的？

使用 LLM 生成网站涉及许多方面。此类别中的大多数产品都遵循类似的（高度简化的）架构图：

![架构图](/images/20250212_ai_powered_web_app_builders_rise/architecture.png)

在其核心，LLM 根据来自 UI 的用户输入生成代码示例。然后，生成的内容通过中间件逻辑进行跟踪，该逻辑跟踪文件、代码更改和第三方 API 调用。这通常是一个长时间运行的系统进程 —— 我们已经看到像 Inngest 这样的代理运行时被采用为技术栈的一部分。

该系统最关键的部分之一是第三方组件调用。就像人类开发人员一样，编码代理需要利用现有的打包库来发送电子邮件、在数据库中存储状态数据、缓存数据或添加身份验证组件。但与人类不同的是，人类参考文档来发现新组件，而编码代理参考 llms.txt 来发现使用工具的方法。

## 为什么这些产品现在才流行起来？

在大量高质量编码和推理数据上训练的模型使得生成可执行代码成为可能。鉴于 JavaScript 和 TypeScript 在 Web 开发中的广泛使用，它们主导了公开可用应用程序代码的分布。因此，这些模型特别适合为 Web 生成应用程序代码。

Web 框架的成熟也使得代理能够更快地创建新元素。浏览器运行时是网站生成产品的天然轻量级"环境"，而不是需要更多计算和服务之间状态管理的重型分布式应用程序。在过去的几年里，应用程序的业务逻辑也进一步转移到客户端，使得在浏览器中运行所有内容变得更加容易。

因此，渲染和测试生成的应用程序变得更加容易。例如，可以生成 Vite + React 应用程序并实时显示给最终用户。在许多方面，网站生成器已经演变为动态的、不确定性较低的脚手架，它们始终提供意想不到但有用的结果，帮助用户更接近他们想要的应用程序。

最后，来自 Resend、Clerk、Supabase、Upstash 等公司的制作精良的库和 SDK 现在支持长尾集成需求。要求 LLM 从头生成身份验证系统或电子邮件发送 API 时，错误率很高。但是，如果我们将任务重新定义为选择和集成现有库，而不是从头开始构建所有内容，那么解决方案就会变得更快、更可靠。

## 用户如何决定使用哪些工具？

我们流量了数千篇关于文本到 Web 应用工具的帖子，并与数十位客户进行了交谈，从高技术开发人员到日常消费者。

我们制作了一个简单的流程图来说明我们的发现 —— 从用户希望从哪里开始的问题开始。

![用户选择工具的流程图](/images/20250212_ai_powered_web_app_builders_rise/user_flow.png)

如果你是一个希望从代码开始的技术用户，你可能只会使用 AI 驱动的代码生成和编辑工具，如 Cursor 或 Windsurf。但如果你想从文本提示开始，你可能会探索另一组用于网站或 Web 应用生成的工具。

值得一提的是，此图表包含一类我们没有深入探讨的产品：设计优先的 UI 生成器。像 Bolt 和 Lovable 这样的工具使你能够通过迭代文本提示来编辑设计，但无法实现像 Relume 或 Uizard 那样的细粒度控制（例如移动元素）。

最后，许多技术用户最终会结合使用这些产品。他们可能会首先在像 Lovable 这样的产品中生成一个基本的 Web 应用，将代码导出到 Cursor，并从更面向设计的工具（如 [21st](https://21st.dev/)）中添加组件。或者，他们可能会从像 Magic Patterns 这样的设计产品开始，然后将代码移动到像 Bolt 这样的工具（[示例](https://x.com/donvito/status/1880543692131553751)）。

## 它们好用吗？

对于以前由于缺乏编码技能而从未能够部署应用程序的用户来说，这些产品可能会感觉像魔法一样。技术用户也很欣赏能够快速上手并将数小时的开发时间简化为几个简单提示的能力。

看到人们赞美这些产品并不罕见：

![用户分享](/images/20250212_ai_powered_web_app_builders_rise/user_positive_feedback.png)

然而，它们今天能够可靠地生成的内容是有限的 —— 这相当于与一位初级开发人员合作，他了解基础知识，但很容易陷入错误的"绝望循环"，特别是当项目变得更加复杂时。

像上传照片到你的网站顶部这样简单的事情可能会完美运行，或者它可能会触发一系列错误，需要你花费 45 分钟来调试。这些产品的好处在于你可以指出错误并要求生成修复，但这可能需要一段时间并需要多种方法！

![发生错误](/images/20250212_ai_powered_web_app_builders_rise/TillyTracker.png)

为什么调试网站生成的代理很难？首先，找到正确的代码进行调试是一个推理和搜索问题：如何定位正确的业务逻辑片段需要代理实现来定位要调试的正确文件/代码行。这并不总是确定性地发生，特别是当通过模型完全加载长上下文时。

另一个原因是代码具有深层依赖关系。系统一部分的更改可能会以不可预测的方式跨多个层级联，并且当这些依赖关系引入异步执行、竞争条件或其他副作用时，调试（尤其是对于 LLM 代理）变得更加困难。

一些常见问题：

![常见问题](/images/20250212_ai_powered_web_app_builders_rise/common_issues.png)

1. ** 集成很困难。** 特别是对于非技术用户来说，添加数据库、身份验证和支付等功能可能会导致重大问题。有很多潜在的故障点 —— 从在另一个服务（例如 Stripe、Supabase）上设置帐户，到正确插入 API 密钥，再到故障排除，特别是如果另一方的流程或文档发生变化。

2. ** 错误持续存在。** 陷入一个令人沮丧的无尽错误循环并不罕见 —— 尤其是当一个开朗的 AI 助手反复告诉你错误已解决（而实际上并非如此）时。为了摆脱这种"绝望循环"，这些工具有时需要关于如何解决问题的额外指导，或者你只需指示它们从头开始。

![错误循环](/images/20250212_ai_powered_web_app_builders_rise/error_loop.png)

3. ** 代码对于产品来说可能"太大"。** 当你生成了大量代码，开始超出产品的管理能力时，通常会出现错误。有时产品会直接告诉你你的项目变得太大了 —— 但它可能只是开始降低性能，甚至以不符合你目标的方式删除/重写代码。

因此，我们发现这些工具主要用于大幅降低原型设计的成本 —— 这是当今的一个巨大痛点。在与客户的对话中，我们发现许多人正在测试这些产品，但如果他们计划向大量受众发布网站和或开始收费，则不会切换过来。很难依赖这些工具始终 100%有效，并且通常很难以可预测的方式迭代你所创建的内容。一个看似简单的更改可能会在代码库的其他地方触发一个难以调试的问题。

然而，与其他 AI 生成内容的形式（例如图像、视频、音乐）一样，我们预计会出现陡峭的改进曲线。这些功能仍然令人印象深刻，并且它们每天都在改进。我们预计这些工具将随着它们的成熟而提升价值链。我们很高兴看到一个新的 AI 原生调试流程出现，以解决该领域的独特挑战。

## 人们用文本到 Web 应用工具构建什么？

当我们仔细研究人们如何利用这些工具时，我们发现了基于用户背景的不同用例。它们通常分为三类：消费者、开发人员（包括新手和有经验的）和顾问。

### 消费者

普通消费者第一次能够构建真正的交互式 Web 应用——而不仅仅是登陆页面或基于模板的系统。我们看到热情的采用来自那些终于能够制作以前需要编码能力的东西的人。

他们在建造什么？我们看到了一长串高度个性化的应用程序，满足用户的特定兴趣和需求。今天，这些产品通常相对简单 —— 它们不需要用户身份验证或付款之类的东西 —— 主要用例是娱乐、满足感或基本实用性。

### 开发人员（包括新手和有经验的）

有各种各样的开发人员——从在工作中编写代码的人到从高层次理解技术概念的用户。我们发现，对于开发人员或与开发人员相邻的受众来说，网站生成器被视为 10 倍更好的新脚手架。这个群体现在可以通过生成他们自己定制的入门模板来走得更远，而不是运行 `npx create-react-app`，这使得编辑代码的门槛比以前低得多。

就像图像生成器简化了艺术家的创作过程一样，网站生成器解决了开发人员的"空白画布"问题。

几个例子：

一位想免费玩数独游戏的用户用 Bolt 构建了一个数独游戏。

一位初创公司创始人为他们的产品在 v0 中创建了一个专业级音频表。

一个用 Bolt 构建的工具，用于在浏览器中测试和调试 Three.js 作品。

### 顾问/机构

在消费者和开发者之间有一个有趣的用户类别：顾问/机构。这些人通常不是工程师，但被聘请为企业（通常是自由职业者和中小企业）创建网站或 Web 应用。

想象一下被聘请为你当地的餐馆或税务会计师制作网站的人。在生成式 AI 之前，他们可能正在使用像 Squarespace 或 Wix 这样的无代码工具。现在，他们中的一些人正在探索文本到 Web 应用工具，作为一种 (1) 更快地进行原型设计和迭代，以及 (2) 承接更复杂项目的方式。

大多数顾问不会公开分享他们的工作和他们用来制作它的工具——但这里有一些他们可能创建的应用程序类型的示例：

使用 Replit Agent 构建的电子商务商店。

拉各斯一家咖啡店的网站 + 商店。

## 接下来会发生什么？

在这个快速发展的领域，我们几乎每天都能看到进步，团队们夜以继日地工作，以实现下一个突破。关于它如何发展的一些想法：

根据用户角色区分产品。今天，这个领域的大多数产品都是"面向所有人"的——它们服务于所有用户，从最具技术性的开发人员到日常消费者。然而，我们认为理想的产品可能因每个用户角色甚至用例而异。例如，面向消费者的最佳文本到 Web 应用工具可能是移动优先的，有很多"护栏"，并优先与 Instagram 和 Shopify 等平台集成。面向开发人员的最佳工具将看起来非常不同！
有可能开启一个向上游市场发展的新领域。我们已经看到不太懂技术的用户为内部使用生成他们自己的生产力工具。这些工具通常在公司的防火墙内下载和部署，这意味着它们很少对公众可见。然而，鉴于这些工具在工作中使用，网站生成平台有机会添加更多企业功能并向上游市场发展。这可能会与现有的低代码内部工具市场相交。
与常见合作伙伴的打包集成。我们听到的最常见的抱怨之一是很难添加集成。如果你想构建一个供其他人使用的应用程序，这一点尤其重要。你可能需要接受付款、创建用户登录、将信息保存在数据库中或从另一个网站提取内容。我们希望看到产品与他们的客户使用的最常见的工具建立"一键式"集成，使其真正无缝，而不是要求用户注册多个服务、提取 API 密钥和研究文档。还将有针对特定组件或代理在生成应用程序时使用的基本工具的生成工具——例如由 Resend 构建的用于电子邮件的 v0。
像素级设计控制。这些产品通常从默认的设计和美学开始。你可以通过提示来编辑它，但没有像 Figma 这样的产品那样高的控制水平——这对于面向设计的消费者或企业来说是一个问题。我们渴望看到这些产品为那些想要它的人提供更细粒度的设计编辑功能，然后将这些更改转换回代码。
定价清晰度和教育。这个领域的大多数产品都有某种基于使用的定价模型——你支付月度订阅费，但它只包括一定数量的令牌（或积分）。如果你想要更多，你必须购买它们或升级。但是用户在开始时不知道他们将使用多少令牌，这使得很难估计构建网站的成本。围绕浪费的令牌也存在很多挫败感，并且有一个由课程/内容创建者组成的迷你行业，可以帮助新用户弄清楚如何"有效地使用他们的令牌"。我们希望公司可以将这些功能引入内部，并更好地指导用户学习如何高效地进行提示。

### 从长远来看，文本到 Web 应用领域会是什么样子？

还值得问的是，这种能力（从文本提示生成 Web 应用）将保留在独立工具中，还是会集成到用户已经在工作的现有产品中。

新的 Replit 代理就是一个有趣的例子。当 Web 应用生成被打包到 Replit 的现有产品中时，它变得更加容易——在生态系统中启动身份验证和数据库是无缝的，你不需要去找第三方提供商。

我们可以想象，其他平台有兴趣探索这个领域，并且这样做对他们现有的客户群有优势。例如：像 Figma 这样的设计工具可以进入代码领域，使设计人员能够从提示到可编辑的设计界面再到部署的网站。或者 AI 编码工具可以添加具有更友好的消费者 UI 的浏览器界面。

我们也很好奇大型 LLM 厂商是否会决定进入这个领域。随着像 OpenAI 和 Anthropic 这样的公司越来越关注代码，他们可能会添加界面供用户编程甚至部署简单的应用程序。
